## Dependency management

Source of truth for dependencies lives in `requirements.in` (runtime) and
`requirements-dev.in` (test/lint/type tools). The committed `requirements.txt`
is the **lock** — it is regenerated by `pip-tools`, not edited by hand.

### First-time setup

```bash
pip install pip-tools
```

### Locking after a change

Whenever you add, remove, or bump a constraint in `requirements.in`:

```bash
pip-compile --generate-hashes --output-file=requirements.txt requirements.in
pip-compile --generate-hashes --output-file=requirements-dev.txt requirements-dev.in
```

Commit both `*.in` files and the regenerated `*.txt` lockfiles together.

### Installing

Production / Docker:
```bash
pip install --require-hashes -r requirements.txt
```

Local dev (adds test, lint, type, and pip-tools):
```bash
pip install --require-hashes -r requirements.txt -r requirements-dev.txt
```

### Notes

- `--generate-hashes` ensures byte-for-byte reproducibility — without it, a
  yanked or repackaged release on PyPI can silently change build output.
- `requirements-dev.in` constrains to `requirements.txt` (`-c requirements.txt`),
  so dev tools never pull a version of a runtime dep that conflicts with the lock.
- The current `requirements.txt` predates this workflow and still ships dev tools.
  Run the `pip-compile` commands above to regenerate a strict runtime-only lock.
